# 运维手册

## 1. 日常巡检

建议每日巡检以下内容：

1. `GET /api/v1/health` 是否返回 `status=ok`
2. `GET /api/v1/platform/summary` 中环境、版本、Agent 列表是否符合预期
3. 前端首页、`/docs`、`/login`、`/dashboard` 是否可正常打开
4. 数据库连接数、慢查询、磁盘容量是否在阈值内
5. Redis 内存占用、连接数、命中率是否异常

## 2. 监控指标

### 后端

- 请求量
- 2xx / 4xx / 5xx 比例
- 请求耗时 P50 / P95 / P99
- 登录失败次数
- 刷新令牌失效次数
- Agent 成功、失败、超时次数
- 单租户 Agent 并发数

### 基础设施

- CPU / 内存
- 容器重启次数
- 磁盘使用率
- PostgreSQL 活跃连接数
- Redis 内存与淘汰情况

## 3. 日志

- 生产环境建议输出 JSON 结构化日志
- 已脱敏字段包括密码、哈希、认证头、刷新令牌、TOTP 密钥
- 建议区分：
  - 应用访问日志
  - 安全与鉴权日志
  - 审计日志

保留建议：

- 应用日志：30 天
- 审计日志：至少 180 天
- 安全事件：如内部策略要求，可保留 365 天

## 4. 备份

- 小规模环境至少每 6 小时一次逻辑备份
- 每日执行一次恢复演练到预发环境
- 生产环境建议启用 WAL 归档或托管 PITR
- 额外备份：
  - `.env` 或密钥管理配置
  - 反向代理配置
  - 部署脚本与版本记录

## 5. 密钥与凭证轮换

应建立定期轮换机制：

- `JWT_SECRET`
- `JWT_REFRESH_SECRET`
- 数据库密码
- Redis 密码
- 第三方模型供应商 API Key

轮换后动作：

1. 重启相关服务
2. 验证登录、刷新令牌与聊天调用
3. 记录轮换时间、责任人与影响范围

## 6. 事故处理

### 租户级事故

- 可直接将 `Tenant.isActive = false`
- 受影响用户执行 refresh token 全量失效
- 必要时暂时下线相关供应商配置

### 鉴权事故

- 强制轮换 JWT 密钥
- 失效现有 refresh token
- 复核最近登录与失败登录日志

### 上游模型服务故障

- 在 `/settings` 中切换 Base URL 或更换供应商
- 暂时提示前端回退到本地说明模式
- 记录受影响模型与时间窗口

## 7. 容量与扩容

建议优先关注：

- 后端 CPU 是否因 Agent 调度或日志过量升高
- 数据库是否因审计日志增长而出现查询退化
- 前端是否因大量文档与静态页面构建导致发布变慢

扩容顺序建议：

1. 托管数据库与 Redis
2. 后端横向扩容
3. 文档与前端静态资源接入 CDN
4. 引入集中日志、指标和告警

## 8. 发布流程建议

1. 在预发环境运行 `npm run typecheck`
2. 运行 `npm run test`
3. 运行 `npm run build`
4. 验证 `/api/v1/health` 与 `/api/v1/platform/summary`
5. 验证 `/docs`、`/settings`、`/chat`、`/dashboard`
6. 再发布到生产

## 9. 建议告警

- 健康检查连续失败
- 5xx 错误率持续升高
- 登录失败数异常飙升
- Agent 超时率持续升高
- PostgreSQL 磁盘或连接数逼近阈值
- Redis 内存接近上限

## 10. 容灾目标

- 建议 RPO：15 分钟
- 建议 RTO：2 小时

## 11. 常见排查路径

### 文档中心打不开

1. 检查前端部署产物是否包含 `docs/`
2. 检查 `/api/docs` 是否能读取目标文件
3. 检查前端工作目录是否变化

### 设置页保存了 Key，但聊天页还是回退

1. 检查保存的供应商是否和当前模型一致
2. 检查 Base URL 是否正确
3. 检查服务端日志里的上游返回内容

### 控制台读不到后端概览

1. 检查 `/api/v1/platform/summary`
2. 检查 `NEXT_PUBLIC_API_URL`
3. 检查 CORS 配置与反向代理转发头
