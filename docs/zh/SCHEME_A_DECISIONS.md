# 方案 A 决策映射

本仓库基于你提供的方案 A 思路，但为了满足“首版内置 Agent 工具调用”“Docker 自托管”“中英双语”这些硬性要求，对少数决策点做了受控调整。

## 直接沿用的决策

- `数据库`：生产环境以 PostgreSQL 为主。
- `认证系统`：走完整认证链路，而不是只保留 API Key。
- `Agent 对话传输`：保留 SSE 友好的后端接口形态。
- `内容管理`：文档与规范继续使用 Git 管理的 Markdown。
- `模型供应商扩展`：Agent 层保持 provider-agnostic，后续可接 OpenAI、Anthropic、Google、DeepSeek、Mistral、Meta。

## 为满足交付要求而调整的决策

- `部署目标`：默认从 Vercel 调整为 Docker 自托管，因为题目明确要求 Docker 与跨系统部署；前端仍可选部署到 Vercel。
- `国际化范围`：交付版只内置 `zh-CN` 与 `en-US`，因为这是当前明确要求。
- `Agent 工具调用`：原表格第 10 行的 A 选项是“首版不支持，二期添加”，但本项目必须首版支持基础工具，因此升级为首版基础工具调用。

## 为什么这样选

- 保留方案 A 的全栈、可扩展、面向后续模型能力接入的路线。
- 同时满足当前仓库要能立刻本地启动、容器化部署、支持多租户登录和 Agent 的现实要求。

## 如何切换或回滚

- 想回到“首版不带 Agent 工具调用”，可关闭注册表中的内置工具并移除前端 Agent 页面。
- 想切回 “前端 Vercel + API 容器化”，保留 API 与数据库容器部署，只把前端改为 Vercel 构建即可。
- 想进一步切换到 Auth.js v5 完整前端会话模式，可在 Next.js 侧增加 Auth.js 适配层，让 API 成为资源服务器。
